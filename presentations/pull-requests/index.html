The Pull Request Workflow

Traditionally

All developers add their changes straight into the main code
  • Developer A is working on a new feature
  • Developer B is working on a bug fix

What's the problem?


* 8d2165a - re-add the new feature after bug fix removed it <Developer A>
* 56bd1f4 - fixed the bug in the firmware journey <Developer B>
* d47efa3 - updated firmware to make use of the new feature <Developer A>
* b3c3b9e - fixed a bug in the new feature <Developer A>
* 19a99be - fixed the bug in the carousel <Developer B>
* 34c0ce5 - started adding awesome feature <Developer A>

What's the problem?

  • Hard to see when a change was introduced
  • Changes from different topics are intermingled
  • Changes can only be retroactively reviewed

Enter Pull Requests!*

*sort of

What Is A Pull Request?

Hey! I've made some changes! You want them?

How does it help?

  • All changes for one topic are collected
  • All changes are introduced to master in a single change
  • Code entering master can controlled and reviewed

Much better
